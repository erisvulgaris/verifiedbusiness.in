SECTION 0: PLATFORM DECLARATION — READ FIRST, NON-NEGOTIABLE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

This is a multi-city, all-India local business directory platform.

It covers:
- All 28 states and 8 union territories (36 total)
- All 780+ districts
- All 4,000+ cities and towns
- All 19,000+ unique India Post pincodes
- Potentially millions of individual business listings

THIS IS NOT A SINGLE-CITY TOOL.
THIS IS NOT A REGIONAL TOOL.
THIS IS NATIONAL INFRASTRUCTURE.

Every single architectural, database, SEO, performance, URL, and design
decision made during development must be evaluated against this scale
from day one. Retrofitting national scale onto a single-city architecture
is one of the most expensive mistakes in web platform development.
Do not build small and scale later. Build for scale now, launch with less data.

GEOGRAPHIC HIERARCHY OF INDIA (memorize this, it drives everything):

  Level 1: Country        → India
  Level 2: State / UT     → 36 entities (e.g., Andaman and Nicobar Islands)
  Level 3: District       → 780+ (e.g., South Andaman)
  Level 4: City / Town    → 4,000+ (e.g., Port Blair)
  Level 5: Locality/Area  → Variable (e.g., Phoenix Bay)
  Level 6: Pincode        → 19,000+ (e.g., 744101)

Every level is a first-class SEO asset. Every level gets its own page type.
No level is skipped or collapsed into another without explicit justification.

SCALE IMPLICATIONS THAT MUST BE ADDRESSED BEFORE WRITING A SINGLE LINE OF CODE:
- Total indexable pages will exceed 1,000,000 (one million) at full data load
- XML sitemaps have a 50,000 URL limit — sitemap index files are mandatory
- Database queries touching all listings must be paginated at the query level
- No full-table scans in production under any circumstance
- Static generation or aggressive caching is required for all geographic pages
- Search indexing must be handled by a dedicated search layer, not raw DB queries

PRIMARY LANGUAGE: English (Phase 1)
MULTILINGUAL: Declared as Phase 2 — architecture must not block this
COUNTRY CODE: IN
CURRENCY: INR (₹)
PHONE FORMAT: Indian 10-digit mobile, optional +91 prefix
SEARCH ENGINE TARGET: Google (96%+ market share in India)
COMPETITOR AWARENESS: JustDial (DA ~70), Sulekha (DA ~60), IndiaMart (DA ~70),
  TradeIndia, Yellow Pages India — your SEO strategy must acknowledge these
  competitors dominate head terms; your advantage is depth, freshness,
  and hyper-local long-tail coverage they cannot scale to match.

SECTION 1: DATA CONTRACT — SINGLE SOURCE OF TRUTH
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Each business listing is a structured record with the following canonical schema.
All optional fields may be null or empty. The system must handle every null state
gracefully. A listing with only name + category + address + city is a valid,
displayable, indexable page.

FIELD DEFINITIONS:

  id                (string, unique, required)
  name              (string, required)
  category          (string, required)
  subcategory       (string, nullable)
  address           (string, required — street + locality level)
  city              (string, required)
  district          (string, required — must be added to data pipeline)
  state             (string, required)
  pincode           (string, required — 6-digit India Post format)
  phone_numbers     (array of strings, may be empty array)
  email             (string, nullable)
  website           (string, nullable)
  google_map_link   (string, nullable)
  coordinates       (object {lat, lng}, nullable)
  source_url        (string)
  canonical_url     (string, required)
  seo_title         (string, required)
  seo_description   (string, required)
  seo_keywords      (string, nullable)
  opening_hours     (string, nullable)
  payment_methods   (string, nullable)
  features          (array of {feature, description}, may be empty)
  faqs              (array of {question, answer}, may be empty)
  about_text        (string, nullable)

DATA QUALITY REALITY CHECK FOR INDIA:
At India scale, the following null rates are normal and expected.
The system must not appear broken when these fields are missing:
- phone_numbers empty: expected in 30–50% of listings
- email null: expected in 60–80% of listings
- website null: expected in 70–85% of listings
- coordinates null: expected in 40–60% of listings
- google_map_link null: expected in 40–60% of listings
The platform must look complete and trustworthy even with minimal data fields.

DATA NORMALIZATION RULES (apply at ingest, before storage):

  PHONE NUMBER NORMALIZATION:
  - Strip all spaces, dashes, brackets from raw input
  - Accept formats: 9876543210 / +919876543210 / 09876543210 / 91-9876543210
  - Normalize all to: +91XXXXXXXXXX (E.164 format) for storage
  - Display as: +91 98765 43210 (human-readable, formatted)
  - Validate: must be 10 digits after stripping country code and leading zero
  - Reject: numbers that do not pass this validation, log for review
  - Handle arrays: some listings have multiple phone numbers — display all

  ADDRESS NORMALIZATION:
  - Indian addresses are highly inconsistent — do not enforce strict formatting
  - Store raw address string as-is from source
  - Separately store and index: city, district, state, pincode as discrete fields
  - These four fields must always be clean, validated, and structured
  - Never store city/state/pincode inside the address string as the only copy

  PINCODE VALIDATION:
  - Must be exactly 6 digits
  - Must begin with a digit between 1 and 9 (no leading zero)
  - Cross-reference against India Post pincode database where possible
  - Store pincode as string, never integer (leading zeros, though rare, exist)
  - Pincode must be associated with its correct city, district, and state
  - If pincode does not match city/state on record, flag for review

  BUSINESS NAME NORMALIZATION:
  - Store original casing from source
  - Generate slug in lowercase with hyphens: "AUTO SOLUTIONS" → "auto-solutions"
  - Slug must be unique per city — if collision, append id suffix
  - Handle names in regional scripts — store as-is, slug from transliteration

  CATEGORY NORMALIZATION:
  - Maintain a master category taxonomy (defined separately)
  - Map all incoming category strings to canonical taxonomy entries
  - Every listing must have exactly one canonical category
  - Subcategory is optional but must also map to taxonomy if present

  DUPLICATE DETECTION:
  - At India scale, duplicate listings are inevitable and frequent
  - Detect duplicates on ingest using: name similarity + address + pincode
  - Flag suspected duplicates for review — do not auto-delete
  - Canonical listing selection: prefer the record with more non-null fields
  - Merged records must retain all source URLs
  - Display only one canonical listing per unique business

  AUTO-GENERATED CONTENT POLICY:
  - Features and FAQs in the data are system-generated
  - This is a known trust risk (automation bias reversal)
  - NEVER display hedging language to users:
    Forbidden phrases: "may provide", "possibly", "might be",
    "could offer", "perhaps", "we believe", "generally"
  - If data is uncertain → omit the claim entirely
  - Differentiate auto-generated content from business-verified content visually
  - Build a listing claim flow in Phase 2 for businesses to verify/edit their data
  - Display "Last updated: {date}" on every listing without exception

NULL STATE HANDLING RULES (exhaustive):
  - phone_numbers empty → hide call CTA entirely, no empty field shown
  - email null → hide email button entirely
  - website null → hide website button entirely
  - google_map_link null AND coordinates null → hide map section entirely,
    show full address as text only
  - coordinates null but google_map_link exists → link to Google Maps, no embed
  - opening_hours null → hide hours section entirely
  - payment_methods null → hide payment section entirely
  - features array empty → hide features section AND its H2 heading
  - faqs array empty → hide FAQ section AND its H2 heading
  - about_text null → hide about section AND its H2 heading
  - All contact buttons null → replace sticky CTA bar with address bar only
  RULE: Never render a section heading without content beneath it.
  RULE: Never render an empty field, dash, "N/A", or placeholder for missing data.
  RULE: A page with only name + category + address must look complete, not broken.

SECTION 2: INFORMATION ARCHITECTURE & URL STRUCTURE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

PAGE TYPE HIERARCHY:

  TYPE 1:  HOMEPAGE                  /
  TYPE 2:  STATE PAGE                /{state-slug}/
  TYPE 3:  DISTRICT PAGE             /{state-slug}/{district-slug}/
  TYPE 4:  CITY PAGE                 /{city-slug}/
  TYPE 5:  CATEGORY PAGE             /{city-slug}/{category-slug}/
  TYPE 6:  SUBCATEGORY PAGE          /{city-slug}/{category-slug}/{sub-slug}/
  TYPE 7:  PINCODE PAGE              /pincode/{pincode}/
  TYPE 8:  BUSINESS DETAIL PAGE      /{listing-id}/{business-slug}/
  TYPE 9:  SEARCH RESULTS PAGE       /search?q=...&city=...&category=...
  TYPE 10: NEAR ME PAGE              /{city-slug}/near-me/
  TYPE 11: TAG PAGE                  /tag/{tag-slug}/

  WHY DISTRICT PAGES MATTER:
  District-level pages target real search queries:
  "businesses in South Andaman District"
  "auto parts shops in South Andaman"
  These are low-competition queries that established competitors do not serve
  well. District pages are a structural SEO advantage — build them.

URL CONSTRUCTION RULES:

  SLUG RULES:
  - All slugs: lowercase, hyphen-separated, ASCII-safe
  - No special characters, no underscores, no dots
  - No trailing slashes inconsistency — decide one standard and enforce it
    (recommended: trailing slash on all directory pages, no trailing slash
    on detail pages — match this to your canonical tags exactly)
  - Slugs must be stable — changing a slug breaks inbound links and rankings
  - State slug: "andaman-and-nicobar-islands"
  - District slug: "south-andaman"
  - City slug: "port-blair"
  - Category slug: "auto-parts-shops"
  - Business slug: generated from name, collision-safe with id suffix

  CITY vs STATE SLUG COLLISION RULE:
  Some city names match state names or district names in India.
  All city slugs must be globally unique in the URL namespace.
  Resolution strategy: append district or state suffix when collision detected.
  Example: if "aurangabad" exists in both Maharashtra and Bihar,
  use "aurangabad-maharashtra" and "aurangabad-bihar"

  CANONICAL URL RULES:
  - Every page has exactly one canonical URL
  - Canonical tag present on every page, pointing to itself
  - Paginated pages: page 1 is canonical for the sequence
  - All URL parameters (filters, sort) use ?param=value format
  - Filter/sort URLs are NOT canonical — canonical always points to base URL
  - www vs non-www: pick one, redirect the other, canonicalize to your choice
  - HTTP must redirect to HTTPS on all URLs, always

INTERNAL LINKING RULES:
  - Every state page links to all its districts and top cities
  - Every district page links to all its cities
  - Every city page links to all its categories with listing counts
  - Every category page links to its subcategories and sibling categories
  - Every listing page links to: city page, category page, 5–8 related listings
  - Every listing page must display breadcrumbs as functional HTML links
  - No page more than 4 clicks from homepage
  - Footer must contain sitemap-style links to all states and top categories
    These must be real <a href=""> HTML links, never JavaScript-only

BREADCRUMB RULES (Information Scent — Pirolli & Card, 1999):
  Breadcrumbs are not decorative. They serve three simultaneous functions:
  1. Navigation: user knows where they are and can backtrack
  2. Information scent: user knows path to their goal
  3. Structured data: BreadcrumbList schema signals hierarchy to Google

  Format per page type:
  State:      Home > Andaman and Nicobar Islands
  District:   Home > Andaman and Nicobar Islands > South Andaman
  City:       Home > Andaman and Nicobar Islands > Port Blair
  Category:   Home > Port Blair > Auto Parts Shops
  Listing:    Home > Port Blair > Auto Parts Shops > AUTO SOLUTIONS
  Pincode:    Home > Port Blair > Pincode 744101

  RULE: Breadcrumbs are HIGH PRIORITY, not nice-to-have.
  RULE: Must be implemented as BreadcrumbList schema simultaneously.
  RULE: Every non-homepage page must have breadcrumbs.

SITEMAP STRATEGY (mandatory at India scale):
  Google's sitemap limit: 50,000 URLs and 50MB per sitemap file.
  At millions of pages, you need a sitemap index file.

  SITEMAP INDEX: /sitemap.xml
    → /sitemap-states.xml         (36 URLs)
    → /sitemap-districts.xml      (780 URLs)
    → /sitemap-cities.xml         (4,000+ URLs)
    → /sitemap-categories.xml     (city × category combinations)
    → /sitemap-pincodes.xml       (19,000+ URLs)
    → /sitemap-listings-001.xml   (up to 50,000 listings)
    → /sitemap-listings-002.xml   (next 50,000)
    → /sitemap-listings-NNN.xml   (continue as needed)

  Each sitemap file must include <lastmod> tag.
  Update sitemap files incrementally on new listing addition.
  Submit sitemap index to Google Search Console on launch.
  Ping Google on sitemap updates: 
    https://www.google.com/ping?sitemap=https://yourdomain.com/sitemap.xml

SECTION 3: USER PSYCHOLOGY & BEHAVIOR MODEL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

CRITICAL PRINCIPLE: Users do not browse this platform for entertainment.
They arrive with varying levels of urgency and intent. Every design decision
must serve their goal, not the platform's desire to display information.

THE THREE COGNITIVE USER MODES (Marchionini, 1995):

  MODE 1 — KNOWN-ITEM SEEKING (High Intent, ~40% of users)
    Internal state: "I need an auto parts shop in Port Blair right now"
    Behavior: Lands deep via Google, needs address/phone in under 10 seconds
    Design response: Contact info above fold, sticky CTA bar on mobile
    Failure state: User cannot find phone number → exits → goes to competitor
    Success metric: Time to first contact action < 10 seconds

  MODE 2 — EXPLORATORY SEEKING (Medium Intent, ~35% of users)
    Internal state: "What options are available for auto parts in Port Blair?"
    Behavior: Scans category pages, compares 3–5 listings, reads details
    Design response: Rich scannable listing cards, clear differentiation between
    listings, easy navigation between results
    Success metric: User views 3+ listings before making decision or exiting

  MODE 3 — DISCOVERY (Low/No Intent, ~25% of users)
    Internal state: "I wonder what businesses are in my area"
    Behavior: Follows links, browses categories, no specific goal
    Design response: Popular/trending sections, "near me" features,
    related listings, category exploration paths
    Success metric: Session depth > 4 pages, return visit

  THE INTERFACE MUST SERVE ALL THREE MODES SIMULTANEOUSLY.
  Do not design only for Mode 1 (the most obvious use case).

SYSTEM 1 vs SYSTEM 2 THINKING (Kahneman, 2011):
  Directory users operate primarily in System 1 — fast, automatic, heuristic.
  They are not reading carefully. They are scanning and pattern-matching.
  Design implications:
  - Use familiar, conventional layouts — novelty increases cognitive friction
  - High contrast, large text, simple language — zero friction
  - Format hours as "9 AM – 6 PM" not "09:00:00 – 18:00:00"
  - Format addresses in natural reading order, not database order
  - Category names must be plain language ("Auto Parts Shops" not "Automotive
    Components Retail Establishments")
  - Round numbers, simple words, conventional icons throughout

COGNITIVE LOAD (Sweller, 1988):
  Three types of load to manage:
  - Intrinsic load = complexity of the business data itself (unavoidable)
  - Extraneous load = unnecessary UI complexity → MINIMIZE AGGRESSIVELY
  - Germane load = pattern recognition, mental model building → SUPPORT THIS

  Rules:
  - Never display more than 7 pieces of distinct information simultaneously
    in one visual unit
  - Group related information into visual chunks (Gestalt proximity)
  - Progressive disclosure: show less on listing cards, more on detail pages
  - Each page type has one primary purpose — do not mix purposes

HICK'S LAW — Decision Time:
  Formula: RT = a + b × log₂(n + 1)
  Every additional choice increases decision time logarithmically.
  - Homepage categories visible without scrolling: maximum 8
  - Primary navigation items: maximum 6
  - Filter dimensions visible simultaneously: maximum 5
  - CTAs visible on any single view: maximum 3
  - Never present unlimited options without grouping or progressive reveal

FITTS'S LAW — Target Acquisition:
  Formula: MT = a + b × log₂(2D/W)
  Smaller and farther targets take exponentially longer to tap accurately.
  - All interactive elements: minimum 44×44 CSS pixels (logical pixels)
  - Primary CTAs (Call, Directions): minimum 48×48 CSS pixels
  - Mobile thumb zone: bottom 40% of screen = easiest reach
  - Primary CTAs must be in the thumb zone on mobile
  - Never place primary actions in top corners on mobile screens

SERIAL POSITION EFFECT — Primacy & Recency:
  Users remember the first and last items in any list best.
  - First listing on a category page receives disproportionate attention
  - Last listing on a paginated page is also recalled more than middle items
  - Apply this to monetization logic: first position has highest value
  - Never randomize listing order without a clear documented rationale
  - Default sort order must be defensible and consistent

TRUST PSYCHOLOGY (McKnight et al., 2002):
  Users are making real-world decisions: visiting physical locations,
  calling phone numbers, spending money. Trust is functionally required.
  
  Required trust signals per listing:
  → "Last updated: {date}" — always visible, never hidden
  → Data source indicator (where this data came from)
  → Verified badge when phone/address has been confirmed
  → Distinction between auto-generated and business-verified content
  → Clear indication if a listing has been claimed by the business owner

  AUTOMATION BIAS REVERSAL — CRITICAL RISK:
  When users detect that content is auto-generated (and they will detect it),
  they stop trusting ALL content on the page — including real, accurate data
  like the address and phone number. This is not a minor UX issue.
  This is a conversion-killing trust failure.
  
  Mandatory countermeasures:
  - Never use hedging language in auto-generated content that users will see
  - Never write "AUTO SOLUTIONS may provide value-added services"
  - If a claim cannot be verified, do not make it
  - Visually mark auto-generated sections differently from verified data
  - Build a listing claim flow so businesses can replace generated content
  - A listing that is claimed by its owner should look different from
    an unclaimed listing

INFORMATION FORAGING (Pirolli & Card, 1999):
  Users follow "information scent" — they click only when the link clearly
  leads closer to their goal. Weak scent = user abandons.
  - Every link must communicate exactly what the user will find after clicking
  - Forbidden CTAs: "Learn More", "Click Here", "Read More"
  - Required CTAs: "See all Auto Parts Shops in Port Blair",
    "View Address and Contact Details", "Get Directions"
  - Breadcrumbs maintain scent for deep-landing users from search engines
  - Category names in navigation must match user's mental model, not
    internal taxonomy naming conventions

PARADOX OF CHOICE (Schwartz, 2004):
  Beyond 6–8 visible options, decision paralysis measurably increases.
  - Category grid on homepage: maximum 8 visible, "Browse All" for the rest
  - Filter options: group into collapsible sections if more than 5 options
  - Related listings section: show 4–6, never more than 8

AESTHETIC-USABILITY EFFECT (Kurosu & Kashimura, 1995):
  Users perceive visually clean interfaces as more functional and trustworthy,
  even when functionally identical to cluttered ones.
  A listing with 3 complete, cleanly rendered fields outperforms a listing
  with 10 fields where 7 are empty, dashed, or showing "N/A".
  Every empty field shown destroys perceived quality disproportionately.

SECTION 4: VISUAL HIERARCHY & SCANNING SCIENCE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

EYE-TRACKING & SCANNING PATTERNS (Nielsen, 2006):
  Users scan listing cards and search results in an F-pattern:
  - Full horizontal scan across the top (business name)
  - Partial horizontal scan across the middle (address/category)
  - Vertical scan down the left edge (icons/labels)
  
  Design implications:
  - Business name: TOP LEFT — highest attention zone — always
  - CTA button: TOP RIGHT on desktop, FULL-WIDTH BOTTOM on mobile
  - Secondary info: middle row, left-anchored
  - Icons: left edge, consistent position across all cards
  - Never place critical information in bottom-right of a card
  - CTA on bottom-right of listing card is one of the most common
    directory design mistakes — it is in the lowest attention zone

PRE-ATTENTIVE PROCESSING (Treisman & Gelade, 1980):
  The brain processes color, size, shape, and contrast in under 250ms,
  before conscious attention begins. Use these attributes deliberately:
  - COLOR: Assign a consistent color to each major category
    Use color to identify category at a glance — not to decorate
  - SIZE: Business name largest, category medium, metadata smallest
  - CONTRAST: Primary CTA must have highest contrast ratio on page
  - SHAPE: Category icons must be distinct shapes, not just color variations
  - RULE: Never use color alone to convey meaning (accessibility)
  - RULE: Icon + text label always (icon alone is ambiguous)

WHITESPACE (Ling & van Schaik, 2002):
  Research shows whitespace increases reading comprehension by up to 20%.
  Whitespace is not empty space — it is an active comprehension tool.
  - Minimum 16px padding inside all card components
  - Minimum 24px gap between listing cards in a grid
  - Minimum 32px vertical gap between sections on detail pages
  - Minimum line height: 1.5× the font size for all body text
  - Section headings: 40px+ space above, 16px+ space below

HEADING HIERARCHY (strict, one H1 per page):
  - H1: Primary subject of the page (exactly one per page)
        → Homepage: Platform name/tagline
        → City page: "Businesses in Port Blair"
        → Category page: "Auto Parts Shops in Port Blair"
        → Listing page: "AUTO SOLUTIONS"
  - H2: Major sections (About, Contact, Features, FAQs, Related Listings)
  - H3: FAQ questions, Feature titles, subsection content
  - H4+: Avoid in directory context
  - Heading hierarchy must be semantically correct for both screen readers
    and search engine crawlers — do not use headings for visual styling

TYPOGRAPHY RULES:
  - Body text: 16px minimum on mobile, 15px minimum on desktop
  - Never use font weight below 400 for body text
  - Maximum line length: 65–75 characters for body paragraphs
  - Category and metadata labels: 12–14px, medium weight (500), muted color
  - Business name on detail page: 22–28px, bold (700), high contrast
  - Business name on listing card: 16–18px, semi-bold (600)
  - Font choice: system fonts or a single well-loaded web font
  - Never load more than 2 font families in total

COLOR & CONTRAST:
  - Body text on white background: minimum 4.5:1 contrast ratio (WCAG AA)
  - Large text (18px+ or 14px+ bold): minimum 3:1 contrast ratio
  - CTA buttons: minimum 4.5:1 contrast ratio, higher preferred

ERROR AND EMPTY STATE SPECIFICATIONS:

SECTION 6: SEO SPECIFICATIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

COMPETITIVE CONTEXT — INDIA-SPECIFIC:
  Your primary competitors and their approximate domain authority:
  - JustDial: DA ~70 — strong in categories and city pages
  - Sulekha: DA ~60 — strong in services and home categories
  - IndiaMart: DA ~70 — strong in B2B and product categories
  - IndiaYellowPages: DA ~50

  HEAD TERM STRATEGY:
  You will not outrank JustDial for "auto parts shop Mumbai" on day one.
  Do not compete on head terms in the first 12 months.
  Your SEO strategy must be:
  - Depth over breadth: cover every city, every pincode, every subcategory
  - Long-tail domination: "auto parts shops in Phoenix Bay Port Blair 744101"
  - Freshness: regular data updates, "last updated" signals
  - Technical excellence: faster load times, better schema, better UX
  - Content uniqueness: unique page content at every level of geography
  
  Mid-tail target pattern (high volume, medium competition):
  "{Category} in {City}" — build and optimize every combination
  
  Long-tail target pattern (low volume, low competition, high intent):
  "{Business Name} {City}" — own the brand query for every listing
  "{Category} in {Area/Locality} {City}" — locality-level pages

SCHEMA MARKUP — REQUIRED:

  On every Business Detail Page:
  {
    "@context": "https://schema.org",
    "@type": "{MappedSpecificType}",
    "name": "{name}",
    "address": {
      "@type": "PostalAddress",
      "streetAddress": "{address}",
      "addressLocality": "{city}",
      "addressRegion": "{state}",
      "postalCode": "{pincode}",
      "addressCountry": "IN"
    },
    "geo": {                          // Include ONLY if coordinates non-null
      "@type": "GeoCoordinates",
      "latitude": "{lat}",
      "longitude": "{lng}"
    },
    "telephone": "{phone}",           // Include ONLY if phone_numbers non-empty
    "email": "{email}",               // Include ONLY if email non-null
    "url": "{website}",               // Include ONLY if website non-null
    "openingHours": "{formatted}",    // Include ONLY if opening_hours non-null
                                      // Format MUST be: "Mo-Fr 09:00-18:00"
                                      // If closing time unknown: omit entirely
                                      // Incomplete time strings are invalid schema
    "paymentAccepted": "{methods}",   // Include ONLY if payment_methods non-null
    "sameAs": ["{google_map_link}"]   // Include ONLY if google_map_link non-null
  }

  SCHEMA TYPE MAPPING:
  Build a complete mapping table from your category taxonomy to schema.org types.
  Examples:
  "Auto Parts Shops"   → AutoPartsStore
  "Restaurant"         → Restaurant
  "Hospital"           → Hospital
  "Pharmacy"           → Pharmacy
  "Hotel"              → Hotel
  "Bank"               → BankOrCreditUnion
  "School"             → School
  Default fallback: LocalBusiness (only when no specific type fits)
  This table must be built before development begins.
  Using generic LocalBusiness when a specific type exists reduces schema value.

  Additional schema types required:
  - BreadcrumbList: every page with breadcrumbs
  - FAQPage + Question + Answer: listing pages with non-empty faqs
  - WebSite + SearchAction: homepage (Sitelinks Searchbox)
  - ItemList: category pages and search result pages

  FAQ SCHEMA REALITY NOTE:
  Google significantly reduced FAQ rich result display after 2023.
  FAQ rich snippets now appear primarily for government and health websites.
  For a business directory, do not expect FAQ schema to consistently
  trigger rich results in SERPs.
  The real value of FAQs is: semantic keyword coverage, NLP entity signals,
  and increased dwell time — not direct rich result appearance.

META TAGS — RULES:
  Title tag: use seo_title field value exactly as provided
  Meta description: use seo_description field value exactly as provided
  Meta keywords: DO NOT implement. Deprecated since 2009, ignored by Google.
    The seo_keywords field is for internal search use only, not page meta.
  Robots: index, follow on all content pages
  Robots: noindex on: /search? pages, /search?page=N pages,
    filtered URLs with parameters, thin pages with < 3 listings

OPEN GRAPH TAGS (required for social sharing):
  og:title → same as page title tag
  og:description → same as meta description
  og:url → canonical URL of page
  og:type → "place" for listing pages, "website" for others
  og:locale → "en_IN"
  og:site_name → platform name

CANONICAL TAG RULES:
  - Self-canonical on every page without exception
  - Paginated pages: each page is self-canonical
    (Google handles pagination via content signals, not rel=next/prev)
  - Filter/sort URLs: canonical points to base unflitered URL
    e.g. /port-blair/auto-parts-shops/?sort=name canonicalizes to
         /port-blair/auto-parts-shops/
  - www and non-www: canonicalize to one, redirect the other
  - Trailing slash consistency: pick one standard, apply everywhere
  - Never include changing parameters in canonical URLs

CRAWLABILITY RULES:
  - All primary navigation links: standard <a href=""> HTML
  - JavaScript-rendered navigation: not acceptable for indexing-critical links
  - robots.txt:
    Allow: all content pages (state, district, city, category, listing, pincode)
    Disallow: /search (search result pages — too many parameter combinations)
    Disallow: /search?* (parameterized search pages)
    Allow: /sitemap*.xml (explicitly allowed)
  - No noindex on any content page with 3 or more listings
  - Pages with 1–2 listings: evaluate for noindex vs. allowing to index
  - The crawl budget at national scale is a real constraint
    Help Googlebot by: fast server, clear sitemap, good internal linking

NAP CONSISTENCY — NON-NEGOTIABLE:
  NAP = Name, Address, Phone
  These three values must be character-for-character identical:
  - On the listing detail page
  - In the schema markup
  - In any other location they appear on the platform
  Store canonical NAP strings in the data layer.
  Never format them differently on different pages.
  Even minor variations ("St." vs "Street") confuse crawlers and
  reduce local SEO effectiveness.

SECTION 7: PERFORMANCE SPECIFICATIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

INDIA-SPECIFIC PERFORMANCE CONTEXT:
  This platform primarily serves users in India on mobile devices.
  Network conditions and devices vary dramatically by tier of city:
  - Metro cities (Mumbai, Delhi, Bangalore): 4G/5G, mid-to-high range devices
  - Tier-2 cities (Lucknow, Jaipur, Bhopal): 4G, mixed device quality
  - Tier-3+ cities and islands (Port Blair): 3G/4G, older/low-end devices,
    slower speeds, higher data-cost sensitivity
  
  Design for the worst realistic case, not the best case.
  A user on a 10 Mbps connection in Port Blair on a budget Android device
  is a primary user of this platform. Their experience must be acceptable.

CORE WEB VITALS TARGETS (Google 2024 thresholds — not older 2-second rule):
  LCP (Largest Contentful Paint):
    Good: < 2.5 seconds
    Needs improvement: 2.5–4.0 seconds
    Poor: > 4.0 seconds
    → Optimize: ensure LCP element (business name or hero image) loads first
    → Preload critical resources, avoid LCP being a lazy-loaded image

  INP (Interaction to Next Paint) — REPLACED FID in March 2024:
    Good: < 200ms
    Needs improvement: 200–500ms
    Poor: > 500ms
    → Optimize: minimize JavaScript execution on interaction
    → Do not block main thread with heavy scripts during user interaction

  CLS (Cumulative Layout Shift):
    Good: < 0.1
    Needs improvement: 0.1–0.25
    Poor: > 0.25
    → CRITICAL FOR DIRECTORY SITES: CLS is the most commonly failed metric
    → Primary causes to eliminate:
      - Images loading without defined width/height attributes
      - Map embeds loading asynchronously and pushing content down
      - Ad units loading and shifting layout
      - Fonts causing text reflow (use font-display: swap)
    → Every image must have explicit width and height set in HTML
    → Every async-loaded block (map, ad) must have reserved space in CSS

PAGE WEIGHT TARGETS:
  Listing detail page total: < 500KB (uncompressed HTML + CSS + JS)
  Category page total: < 400KB
  Homepage total: < 600KB
  
  Images: serve WebP with JPEG fallback, AVIF where supported
  Never load full-resolution images — serve appropriately sized variants
  Use responsive images (srcset) for all listing images

ADDITIONAL PERFORMANCE RULES:
  - Lazy load all content below the fold: images, map embeds, related listings
  - Never embed map iframes on mobile — use static image + tap-to-open link
  - Paginate at query level: never fetch all listings then paginate client-side
  - Dedicate a search index (e.g., Elasticsearch, Meilisearch, Typesense)
    for text search — never run LIKE queries on the primary database
  - Cache all geographic pages aggressively (state, district, city, category)
    These pages change infrequently — CDN caching with long TTLs is appropriate
  - Listing detail pages: cache with shorter TTL or on-demand invalidation
  - Serve all pages over HTTPS
  - Enable HTTP/2 or HTTP/3 on server
  - Compress all text responses (gzip or brotli)
  - Critical CSS inlined, non-critical CSS deferred
  - JavaScript: defer all non-critical scripts
  - Above-the-fold content must render without JavaScript execution

LOADING STATE RULES:
  - Skeleton screens preferred over spinners for content areas
  - Skeleton shape must match the content it represents (not generic boxes)
  - Maximum loading indicator display: 3 seconds
  - If content is not ready in 3 seconds: show partial content +
    continue loading remaining sections below
  - Never show a fully blank page to a user at any point

PAYMENT METHOD DISPLAY:
  Indian payment ecosystem is unique. Display payment methods with
  India-specific context and icons where possible:
  - UPI (PhonePe, Google Pay, Paytm, BHIM) — most important in India
  - Cash — still dominant in tier-2/3 cities
  - Credit/Debit Card
  - Net Banking
  - EMI
  
  If payment_methods field contains "Cash accepted":
  → Display: "💵 Cash Accepted"
  Also display: "Confirm UPI and card acceptance directly"
  
  UPI acceptance is a significant trust signal in India.
  Seeing "UPI Accepted" reduces hesitation for Indian users.
  Prompt businesses to confirm UPI acceptance when they claim their listing.

PHONE NUMBER DISPLAY:
  Raw display format: +91 98765 43210
  Click behavior: tel:+919876543210 (E.164 format, no spaces)
  If multiple phone numbers: show all, with "Primary" label on first
  WhatsApp: if number is likely on WhatsApp, offer WhatsApp link
  (wa.me/+91XXXXXXXXXX) — WhatsApp is heavily used for business in India

REGIONAL AND LANGUAGE CONSIDERATIONS (Phase 1 — English only):
  Even in an English-only Phase 1, accommodate:
  - City name aliases: "Bombay" → Mumbai, "Madras" → Chennai,
    "Calcutta" → Kolkata, "Bangalore" → Bengaluru
    Store aliases in city table, resolve search to canonical city name
  - Business names in regional scripts: store and display as-is
    Slug generation must use transliteration, not character stripping
  - Hindi transliteration for search: "auto parts ki dukan" should surface
    "Auto Parts Shops" results (basic keyword matching)
  
  MULTILINGUAL PHASE 2 ARCHITECTURE NOTE:
  The URL structure, database schema, and component architecture must
  not block future multilingual implementation.
  Do not hardcode English strings into database records.
  Separate content from structure from presentation.
  Plan for: Hindi (hi-IN), Tamil (ta-IN), Telugu (te-IN), Marathi (mr-IN)
  as likely Phase 2 languages.
  When Phase 2 comes: hreflang tags will be required on all pages.

INDIA ADDRESS DISPLAY FORMAT:
  Display address in this standard Indian postal format:
  Line 1: {street address}
  Line 2: {locality/area}
  Line 3: {City} – {Pincode}
  Line 4: {District}, {State}

  Example:
  35, Maulana Azad Road
  Phoenix Bay
  Port Blair – 744101
  South Andaman, Andaman and Nicobar Islands

  This matches the format Indian users read and recognize.
  Do not display address as a single unformatted string.

  - Body text: minimum 4.5:1 against background

SECTION 11: CONTENT QUALITY STANDARDS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ABOUT TEXT STANDARDS:
  - Must be factually accurate — no fabricated claims
  - Minimum 60 words, maximum 300 words for detail page
  - Must naturally include: business name, category, city, pincode, hours
  - Must not be keyword-stuffed or obviously templated
  - Must read as natural, human-level prose
  - If about_text is obviously a template with variable substitution and
    generic filler, it must be rewritten before the listing goes live

FAQ CONTENT STANDARDS:
  - Questions must reflect real user search intent
  - Answers must be concise: 2–4 sentences maximum
  - First FAQ: highest-intent question ("Where is {name} located?")
  - If a field is null (e.g., no phone number), the FAQ must not
    pretend the information exists — say "contact them to inquire"
  - Forbidden FAQ answer patterns:
    "They may be able to help you with..."
    "You might want to consider..."
    "This business possibly offers..."
  - If a true answer cannot be given: omit the question entirely

STATE AND CITY PAGE CONTENT:
  - Intro paragraphs must be unique per page
  - Cannot be a template with city/state name swapped in
  - Google treats near-identical paragraphs with swapped place names
    as thin or duplicate content — this will harm rankings
  - Each state and city page intro must contain unique geographic,
    demographic, or commercial context about that specific location

FRESHNESS SIGNALS:
  - "Last updated: {date}" displayed on every listing page
  - If data has not been verified or updated in > 12 months:
    Show a "Help us keep this updated" prompt with correction form
  - Outdated data destroys trust and increases bounce rate
  - Build a data refresh pipeline for high-traffic listings

SECTION 12: ANALYTICS & MEASUREMENT
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

PRINCIPLE: Define success metrics before building features.
A feature without a measurable success criterion is a guess, not a product.

EVENTS TO TRACK ON BUSINESS DETAIL PAGE:
  - Page view (with: city, state, category, listing_id)
  - Phone number click (call intent)
  - Directions click
  - Website click
  - Email click
  - Address copy action
  - FAQ accordion open (which question, position in list)
  - Scroll depth (25%, 50%, 75%, 100%)
  - Time on page / dwell time
  - Related listing click (which listing, which position)
  - Share action

EVENTS TO TRACK ON CATEGORY/SEARCH PAGES:
  - Listing card click (position, listing_id, city, category)
  - Filter applied (which filter, which value)
  - Filter removed
  - Sort order changed
  - Pagination: page number accessed
  - Zero result state triggered (log query + city)

EVENTS TO TRACK ON HOMEPAGE:
  - Search initiated (keyword entered, city entered)
  - Category tile clicked (which category)
  - State clicked
  - City clicked
  - Featured listing clicked

ACTIONABLE USES OF THIS DATA:
  - Listings with high views but zero contact actions → improve content quality
  - Categories with high traffic but low listing count → source more data
  - Zero-result queries → identify content gaps, add missing categories
  - Filters never used → remove or reposition
  - FAQ questions never opened → remove or reorder
  - High bounce on specific cities → investigate data quality for that city
  - Scroll depth drop-off points → content below that point needs improvement

SEARCH CONSOLE MONITORING:
  - Monitor impressions vs. clicks per page type
  - Track Core Web Vitals in Search Console reports
  - Monitor index coverage — detect pages being excluded
  - Track queries driving traffic to identify ranking opportunities
